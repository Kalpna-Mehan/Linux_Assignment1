Question 4 – File Access and I/O Investigation

1. Open Files Identified

Command:
lsof app.log

Output:
COMMAND PID   USER FD   TYPE DEVICE SIZE/OFF  NODE NAME
tail    649 kalpna  6r   REG   8,48        33 35352 app.log

Observation:
The lsof command showed that the process "tail" (PID 649), which was left running in the background reading app.log, currently holds the file open. This confirms that Linux tracks every process actively accessing a file, along with the specific file descriptor being used.

--------------------------------------------------

2. File Descriptor Observations

Output (from the lsof result above):
FD column = 6r  (file descriptor number 6, opened read-only)

Observation:
File descriptor 6 was assigned by the OS to the tail process for reading app.log. File descriptors are small non-negative integers the kernel uses internally to reference open files; 0, 1, and 2 are reserved for stdin, stdout, and stderr respectively, and additional descriptors like 6 here are assigned sequentially as a process opens more files.

--------------------------------------------------

3. Output Redirection and Error Redirection

Commands:
echo "Normal output message" > output.txt
ls nonexistentfile 2> error.txt
cat output.txt
cat error.txt

Output:
$ cat output.txt
Normal output message

$ cat error.txt
ls: cannot access 'nonexistentfile': No such file or directory

Observation:
The > operator redirected standard output (the echo message) into output.txt, while the 2> operator redirected standard error (the "No such file" message from ls) into error.txt. This confirms stdout (fd 1) and stderr (fd 2) are independent streams that can be captured separately.

--------------------------------------------------

4. Resource Limits Observed

Command:
ulimit -a

Output (key lines):
core file size          (blocks, -c) 0
file size               (blocks, -f) unlimited
max locked memory       (kbytes, -l) 65536
open files              (-n) 10240
stack size              (kbytes, -s) 8192
max user processes      (-u) 29253
virtual memory          (kbytes, -v) unlimited

Observation:
The ulimit command displayed the current shell's resource limits. Notably, "open files" is capped at 10240 — meaning a single process (like the tail process above) can hold at most 10,240 file descriptors open simultaneously. These limits exist to prevent any one process from exhausting system resources.

--------------------------------------------------

5. How Linux Manages File I/O

Linux manages file input and output using file descriptors — small integers returned by the kernel when a file is opened, which the process then uses for all subsequent read/write calls instead of the filename itself. Every process starts with three standard descriptors (stdin=0, stdout=1, stderr=2), and shell redirection operators (>, >>, 2>, 2>&1) let a user reroute these streams independently, as demonstrated above with output.txt and error.txt.

--------------------------------------------------

Conclusion

The investigation demonstrated how Linux tracks open files via lsof, assigns and manages file descriptors, redirects standard output and error streams independently, and enforces resource usage through ulimit. Together these mechanisms ensure efficient, controlled, and secure file access across all running processes.
